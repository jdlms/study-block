Repository Review

Study Blocks: Readability & Idiomatic Review

Overall, the repo is already pretty solid: small, coherent, and easy to navigate. The biggest improvement areas are less about bad code and more about concept consistency, splitting multi-purpose files, and tightening a few Go/Svelte idioms.

Overall assessment
Good foundation

Readable and compact, with a few concentrated cleanup opportunities.

Biggest issue
Spec drift

The repo tells slightly different stories in docs vs implementation.

Highest leverage
File separation

Split larger files by concern, especially Telegram and frontend logic.

Top opportunities

These are the most worthwhile improvements from a readability and idiomatic point of view.

High priority

internal/service/telegram.go is doing too much

The file currently mixes transport types, polling, command handling, formatting, and Telegram API calls.

  • Suggested split: telegram_client.go, telegram_handler.go, maybe telegram_format.go.
  • Why: Smaller files with one concern are more idiomatic and easier to skim in Go.
High priority

SPEC.md and implementation have drifted apart

There are multiple concept mismatches that reduce trust in the repo as documentation.

  • Spec says Blocks, code uses Minutes.
  • Spec says chart is stacked; frontend config uses stacked: false.
  • Spec says frontend refreshes every 60s; code refreshes at midnight only.
  • Spec references files that do not exist, like internal/telegram/poller.go.

Why it matters: readability includes being able to trust the docs.

High priority

frontend/src/App.svelte mixes too many concerns

The component currently contains date helpers, responsive logic, localStorage state, fetch logic, chart shaping, and rendering.

  • Suggested split: lib/dates.js, lib/chart.js, lib/colors.js, maybe a small legend component.
  • Why: Svelte components read best when they mostly describe UI state and markup.

Go-specific readability / idiomatic notes

Medium

App owns low-level dependencies directly

App keeps a raw *store.Store mainly so it can close it later. That leaks storage details into the app root.

type App struct {
    server   *http.Server
    store    *store.Store
    telegram *service.TelegramService
}

More idiomatic options would be storing generic closers or lifecycle functions instead.

Medium

Unused metadataBucket adds cognitive load

The store creates a metadata bucket, but nothing uses it. That looks like incomplete functionality and makes the reader wonder what they missed.

  • Either remove it for now, or
  • actually use it for Telegram offset persistence, as hinted by the spec.
Medium

Prefer named API request types over inline anonymous structs

In internal/handler/api.go, the POST request payload is defined inline. That is okay for tiny handlers, but named request types make the API contract easier to find and reuse.

Medium

Separate GET and POST bodies into dedicated handler functions

The method switch inside entries is fine, but the handler would read cleaner if the real logic lived in listEntries and createEntry.

Medium

isLocalRequest is brittle

It matches prefixes against r.RemoteAddr, including "localhost:", which generally will not appear there.

Using net.SplitHostPort and comparing parsed host values would be clearer and more robust.

Low

localURL should use strings.HasPrefix

This:

if addr[0] == ':' { ... }

works because earlier guards protect the empty case, but strings.HasPrefix(addr, ":") expresses intent more clearly.

Medium

Ignored errors in Telegram HTTP request construction

Calls like http.NewRequestWithContext(...) and json.Marshal(...) are used with _ ignored.

Even if failure is rare, explicit handling is more idiomatic and avoids making the reader infer that the error is impossible.

Medium

Telegram polling retries silently

Run swallows polling errors and just sleeps and continues. A small log line would make runtime behavior much easier to understand.

Design note

The Bolt key format implies optimizations the code does not use

ListEntries, delete-by-ID, and undo all scan broadly even though the key format is date-prefixed.

For a small app, that is completely fine. But either using prefix scans or explicitly commenting that full scans are intentional for small datasets would better match reader expectations.

Low

DeleteLatestEntryForDate returning nil, nil is subtle

That API shape is valid, but readers have to remember that “not found” is represented by a nil entry without an error. Returning ErrNotFound or documenting the convention would be clearer.

Frontend readability notes

Medium

window.matchMedia(...) at module init is a little sharp-edged

It is fine for a purely client-rendered app, but less robust if SSR or pre-rendering ever appears. Initializing in onMount would be safer and more idiomatic.

Medium

Reactive declarations are dense

The sequence of derived state in App.svelte is idiomatic Svelte, but a lot of transformation logic is embedded inline.

  • buildTotals(entries)
  • buildDatasets(subjects, dates, totals, ...)
  • buildChartOptions(...)

Helper functions would make the reactive block much easier to scan.

Low

There is repeated date formatting noise

Formatting the same dates repeatedly inside dataset creation makes the logic harder to read. Precomputing todayKey and dateKeys would simplify the chart setup.

Low

The color swatch should probably be a real button

Using a <span role="button" tabindex="0"> works, but a semantic <button> would be easier to read and more idiomatic HTML.

Smaller polish items

Low

Naming could be a touch more explicit

  • allowed in IngestService could become allowedSubjects or subjectSet.
  • fmtDuration could be formatMillis or formatRequestDuration.
  • statusWriter could be responseStatusWriter.
Low

Test setup patterns could be more consistent

API and ingest tests repeat store/subject setup. Shared helpers would reduce repetition. Also, pick one cleanup style consistently; t.Cleanup is generally nicer than mixed use of defer and cleanup hooks.

Low

Doc comments would help as the codebase grows

Even under internal/, exported names like App, Config, Store, and TelegramService benefit from brief doc comments once the repo gets bigger.

Most important consistency issue

Settle on one conceptual unit: blocks or minutes

This is the single biggest vocabulary mismatch in the repo:

  • spec talks about blocks
  • code talks about minutes
  • frontend subtitle talks about study minutes
  • Telegram parsing accepts raw minutes

Pick one concept and use it everywhere.

Recommended priority order

1. Fix spec drift Make docs and code tell the same story.
2. Split telegram.go Separate transport, polling, and command handling.
3. Break up App.svelte Move helpers and chart shaping out of the component.
4. Tighten a few Go APIs isLocalRequest, ignored errors, and not-found semantics.
5. Polish naming and tests Lower priority, but helpful once the bigger issues are done.

Net assessment

The repo is already:

  • small
  • understandable
  • reasonably idiomatic
  • easy to navigate

The main thing holding it back from feeling cleanly idiomatic is concept drift and a few files doing multiple jobs at once, not low-quality code.
